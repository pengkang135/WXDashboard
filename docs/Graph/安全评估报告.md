# WXDashboard 微信安全评估报告

> 评估日期: 2026-05-25 | 版本: 1.0 | 机密等级: 内部

---

## 一、执行摘要

WXDashboard 通过 wx-cli (npm) 读取本地微信 SQLCipher 加密数据库，实现工作群消息归档与检索。**本项目已有两次被微信踢下线的前科。** 当前架构在防御层面做了不少工作（守护进程隔离、频率控制、单入口收敛），但仍存在若干可被微信反作弊系统检测的风险点，以及 AI 辅助操作中的绕过路径。

**总体风险等级: 中高** — 微信对本地数据库访问的检测机制不透明，零容忍可能是现状。

---

## 二、架构与数据流

```
微信进程 (Weixin.exe)
  │  SQLCipher 密钥在内存中
  ▼
wx-cli daemon (Named Pipe: \\.\pipe\wx-cli-daemon)
  │  启动时提取密钥一次，缓存在 daemon 进程内
  │  后续查询使用缓存密钥，不接触微信进程
  ▼
┌────────────────── safe_wx.py ──────────────────┐
│  _run_wx_raw()  ← 无频率限制                     │
│    ├─ wx daemon status                         │
│    ├─ wx daemon start                          │
│    ├─ wx sessions -n N --json                  │
│    └─ wx export <群名> --format json -o tmp     │
└────────────────────────────────────────────────┘
  │
  ▼
┌──────────── sync_engine.py ────────────┐
│  _run_wx()  ← 有频率限制 (WX_MIN_INTERVAL) │
│    └─ wx members <群名> --json           │
│                                        │
│  sync() — 统一入口                      │
│    1. 发现新群 (sessions)               │
│    2. 新群全量同步 (export, 500条)      │
│    3. 已有群增量同步 (export, 200条)    │
│    4. 文件下载 (纯文件系统, 无 wx-cli)   │
└────────────────────────────────────────┘
  │
  ▼
SQLite (data/ledger_v2.db)  ← AI 读取的唯一数据源
  │
  ▼
Flask REST API (:8888) → 浏览器 SPA
```

### 关键设计原则

- **"先拷贝再提取"**: `wx export` 将消息写入临时 JSON 文件，Python 从该文件读取。微信原始 SQLite 文件 (MSG.db) 永不被直接访问。
- **守护进程隔离**: SQLCipher 密钥只存在于 wx-cli daemon 进程内，Python 端不接触密钥。
- **单入口收敛**: 所有同步操作统一到 `sync()` 一个函数，便于审计和限流。

## 三、当前防御机制

### 3.1 频率控制

| 参数 | 默认值 | 环境变量 | 作用域 |
|------|--------|----------|--------|
| `WX_MIN_INTERVAL` | 1.5s | `WX_MIN_INTERVAL` | `_run_wx()` 两次调用最小间隔 |
| `SYNC_DELAY_MIN` | 0.5s | `WX_SYNC_DELAY_MIN` | 群间随机延迟下限 |
| `SYNC_DELAY_MAX` | 2.0s | `WX_SYNC_DELAY_MAX` | 群间随机延迟上限 |
| `SYNC_BATCH_LIMIT` | 200 | `WX_SYNC_BATCH_LIMIT` | 单次 export 最大消息数 |

**缺口**: `safe_wx.py` 中的 `_run_wx_raw()` 不受任何频率限制。daemon 状态检查和 session 列表导出可瞬时连续调用。

### 3.2 AI 操作禁令 (CLAUDE.md)

CLAUDE.md 包含 5 条不可协商规则:

1. 只有 `sync_engine.py` 可调用 wx-cli
2. AI (含子 Agent) 绝对禁止调用 wx-cli
3. AI 数据来源只能是 SQLite
4. 本项目禁止子 Agent
5. PreToolUse hook 自动阻断 Bash 中的 wx 模式

### 3.3 PreToolUse Hook

`C:\Users\Kevin\.claude\hooks\block-wx-cli.py` 拦截 Bash 工具调用中的 18 种 wx 相关模式（`wx export`, `wx-cli`, `MSG.db` 等）。仅豁免 `pip install wxpython` / `pip install python-wx`。

### 3.4 无自动同步

前端 `app.js` 不含任何自动/定时同步逻辑。同步仅由用户手动点击"刷新消息"按钮触发。

## 四、风险矩阵

### 4.1 按攻击向量评估

| 风险向量 | 等级 | 说明 |
|----------|------|------|
| **频率模式检测** | 中 | 群间 0.5-2s 随机延迟，但无每日总量上限。连续同步 50+ 群仍会产生密集的 wx-cli 调用序列 |
| **微信进程交互** | 低 | Daemon 模式只在启动时 attach 一次，后续走缓存。`config.py` 中另有 pymem 内存扫描（用于图片解密），属独立风险 |
| **人为误操作** | 高 | Claude Code 会话中一句误写的 Bash 命令即可直接调用 wx-cli。Hook 覆盖 Bash 工具但不覆盖 MCP shell |
| **API 暴露** | 中 | `POST /api/sync/refresh` 无认证，本地任何进程可触发完整同步链 |
| **工具链绕过** | ~~中高~~ 已修复 | MCP Shell 工具 (`mcp__shell__*`) 现已受 PreToolUse hook 保护 |
| **node 间接调用** | 低 | `sync_engine.py` 检测到 `.cmd` 包装时走 `node wx.js`，Bash 文本中不出现 `wx` 字面量 |

### 4.2 按 wx-cli 子命令评估

| 子命令 | 调用频率 | 风险 | 说明 |
|--------|----------|------|------|
| `wx daemon status` | 每次 sync 1次 | 低 | 轻量，仅检查 Named Pipe |
| `wx daemon start` | 按需 (首次) | 中 | 需 attach 微信进程提取密钥 |
| `wx sessions -n 50` | 每次 sync 1次 | 低 | 只读会话列表 |
| `wx sessions -n 200` | 发现新群时 | 低 | 同上，limit 更大 |
| `wx export <群> -n 200` | 每群 1次 (增量) | 中 | 读取加密 DB，最核心的操作 |
| `wx export <群> -n 500` | 每新群 1次 (全量) | 中 | 同上，limit 更大 |
| `wx members <群> --json` | 每新群 1次 | 低 | 只读群信息 |

### 4.3 一次完整同步的 wx-cli 调用量估算

假设 30 个已跟踪群 + 2 个新群:

| 操作 | 调用次数 |
|------|----------|
| daemon status | 1 |
| sessions (发现) | 1 |
| sessions (增量位置) | 1 |
| members (新群) | 2 |
| export (新群全量) | 2 |
| export (已有群增量) | ~5 (仅位置变化的群) |
| **合计** | **~12 次** |

平均间隔: 约 2-3 秒/次。单次完整同步耗时约 30-60 秒。

## 五、已知绕过路径

### 5.1 MCP Shell 绕过 ✅ 已修复

Hook 已扩展至 `mcp__shell__start_process`、`mcp__shell__execute_shell_command`、`mcp__shell__interact_with_process`。`block-wx-cli.py` 采用递归字段扫描，无论 JSON 结构如何均可检测。

### 5.2 HTTP API 触发

```bash
curl -X POST http://127.0.0.1:8888/api/sync/refresh
```

任何本地进程（浏览器扩展、恶意脚本、其他用户）均可触发完整同步。Hook 不拦截 curl/HTTP 调用。

### 5.3 Python import 链

```python
python -c "from backend.safe_wx import export_and_read_messages; print(export_and_read_messages('某群'))"
```

`safe_wx.py` 中的函数可被任意 Python 代码导入调用，无需经过 `sync_engine.py` 的 `sync()` 限流。

### 5.4 node 间接路径

当 `wx` 命令解析为 `.cmd` 文件时，`sync_engine.py` 自动切换为 `node <wx.js>`。此时 Bash 命令文本不含 `wx` 字面量，Hook 无法匹配。

6. **守护进程健康检查去重**

   当前每次 `sync()` 都执行 `wx daemon status`。可缓存 daemon 状态 30 秒，减少不必要的调用。

8. **只读模式**

   提供"仅浏览"模式：不调用 wx-cli，仅展示 SQLite 中已有数据。适合快速查阅场景。

9. **审计日志增强**

   记录每次 wx-cli 调用的完整参数、耗时、返回值，便于事后分析异常模式。

    监控 wx-cli 调用的失败率、响应时间变化，若出现连续失败或响应变慢，可能是微信反作弊系统开始干预的信号。
