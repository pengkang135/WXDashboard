# 微信工作群关键信息提取

从 WXDashboard 项目群消息中提取结构化信息，写入 `ai_extractions` 表。提取目标不仅是消息中的事实信息，还包括**推断群的专业/供货类别**（根据群名、消息内容判断该群是做什么的）。

**安全约束**: 绝对禁止调用 wx-cli。所有消息从 SQLite 读取。

## 适用场景

- 用户要求提取群消息中的关键信息
- 批量提取所有群的结构化信息
- 更新已有提取结果

## 前置条件

- WXDashboard 项目环境，`backend.database` 可用
- 消息已通过正常同步流程写入 SQLite

## 输入

- 目标群ID列表（可选，默认所有 `total_messages > 0` 的群）
- 项目名称（默认当前活跃项目）

## 输出

- `ai_extractions` 表更新的记录
- 提取统计：总记录数、覆盖群数、各类型分布

## 提取维度

- **联系人**: 姓名、角色/职位、公司、邮箱/电话。姓名+公司+角色缺一不可，缺少任一项则不提取。
- **工期节点**: 里程碑事件、日期、责任方。必须有明确日期或时间范围。
- **技术参数**: 参数名、具体数值/规格/标准。必须包含具体数值或规格。
- **文件引用**: 文件名、文件消息的 msg_date（必须，用于构建文件路径）、用途说明。
- **专业/供货类别**: 推断该群的专业领域或供货品类。根据群名、消息内容判断这个群是做什么的——供应商群提取供货品类（如钢板桩、管道、混凝土），施工群提取专业（如钢结构、消防、电气）。每个群都要提取，无门槛。不确定时标注 `"待确认"`。

## 标准步骤

### Step 1 — 查询群列表

从 SQLite 获取所有有消息的群：

```python
from backend.database import get_group_message_stats
stats = get_group_message_stats()
# 过滤 msg_count > 0 的群，按 msg_count 降序
```

### Step 2 — 顺序逐群处理

**禁止并行。一次只处理一个群。** 按消息量从大到小排序，逐群执行：

```python
from backend.database import get_messages_for_ai_processing

for g in target_groups:
    # 读取该群消息（每次最多200条，从旧到新）
    msgs = get_messages_for_ai_processing(g["id"], max_messages=200)
    # 如消息超过200条，分段读取
    # 提取四维信息
    # 稍作停顿(0.5-1s)再处理下一群
```

### Step 3 — 提取信息

对每条消息按五个维度识别信息：

- **联系人信息**: 姓名 + 公司 + 角色缺一不可，缺少任一项则不提取
- **工期节点**: 必须有明确日期或时间范围
- **技术参数**: 必须包含具体数值或规格
- **文件引用**: 文件名 + 发送日期。必须从消息中获取实际的 `msg_date` 字段填入，用于前端构建文件路径
- **专业/供货类别**: 每个群至少提取一条。从群名和消息内容综合推断该群的专业领域或供货品类。例如群名含"ArcelorMittal"且消息讨论钢材价格 → 类别"钢材供应商"、专业"钢板桩"。无法确定具体专业时类别填 `"待确认"` 并在说明中记录线索

对不确定的信息在值中标注，如 `"待确认: ..."`。
同一群内同类型重复信息只保留一条。

输出格式（直接可写入数据库的元组列表）：

```
Group <id>:
('联系人', '{"姓名":"...", "角色":"...", "公司":"...", "邮箱":"..."}')
('工期节点', '{"节点":"...", "日期":"...", "责任方":"..."}')
('技术参数', '{"参数":"...", "值":"..."}')
('文件引用', '{"文件":"...", "msg_date":"...", "说明":"..."}')
('专业/供货类别', '{"类别":"...", "专业":"...", "说明":"..."}')
```

### Step 4 — 审核

检查提取结果：
- 联系人信息是否完整（姓名+公司+角色）
- 技术参数是否有具体数值
- 日期是否在消息时间范围内
- 质量不足的群补充后再审

### Step 5 — 写入数据库

```python
from backend.extraction_writer import write_extractions

# group_data 格式: {group_id: [(extract_type, content_json), ...], ...}
result = write_extractions(group_data)
# result: {'total': 586, 'groups_count': 39, 'by_type': {'联系人': 219, ...}}
```

### Step 6 — 汇总

输出: 覆盖群数、总记录数、各类型（联系人/工期节点/技术参数/文件引用）分布。

## 检查点

- 所有 `msg_count > 0` 的群都已覆盖
- 每条提取的 `content` 是合法 JSON
- `extract_type` 在五种类型之内
- 同一群内同类型同内容无重复
- **专业/供货类别** 每个群至少有一条，无信息的标注"待确认"
- 处理过程中无 wx-cli 进程产生

## 常见失败

| 失败 | 处理 |
|------|------|
| 某群消息过多（>200条） | 分段读取：先读前200条，如有必要再读后续批次 |
| JSON 中文编码问题 | 写入时使用 `json.dumps(..., ensure_ascii=False)` |
| 数据库写入失败 | 检查外键约束（group_id 必须存在于 groups 表） |
| 某群返回格式不规范 | 根据上下文重新格式化，将自然语言描述转为指定 JSON 结构 |
